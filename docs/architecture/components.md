# Components and their Responsibilities

This document contains all the components from the project architecture, and describes the Responsibilities of each component.

## Components
**File Handling**

- Users upload files from the frontend. Initially, limit uploads to **10 MB**.
- The frontend requests a **presigned upload URL** from the API. The API generates a unique object key and returns a presigned URL. The browser uploads the file directly to the storage bucket, bypassing the backend.
- After the upload succeeds, the frontend submits a processing request to the API with the uploaded object's key and the requested operation.
- The API creates a **Job** record in PostgreSQL with an initial `PENDING` status, then pushes only the `job_id` to the Redis task queue.
- A worker fetches the `job_id` from the queue, updates the job status to `RUNNING`, downloads the input file from the storage bucket, performs the requested processing, uploads the output back to storage, and updates the job status in PostgreSQL.
- Users download the processed file through a short-lived **presigned download URL** generated by the API.

**Workers**

- Workers are independent Python processes that continuously consume jobs from the Redis queue.
- A worker is capable of handling multiple processing operations rather than being dedicated to a single file type or task.
- Workers update job state directly in PostgreSQL (`Pending` →`RUNNING`, `RETRYING`, `COMPLETED`, `FAILED`) and publish job events through Redis Pub/Sub after each state change.

**Job Status & Client Updates**

- The API maintains SSE connections with clients.
- Workers publish lightweight job events (`job_completed`, `job_failed`, `job_retrying`) through Redis Pub/Sub.
- The API subscribes to these events and forwards the updated job status to the appropriate client via SSE, allowing the frontend to display live progress and completion status.

**API, Redis & Database Coordination**

- **PostgreSQL** is the source of truth for all job metadata and state.
- **Redis Queue** is responsible only for distributing work to workers.
- **Redis Pub/Sub** is used only for real-time event notifications between workers and the API.
- The API creates new jobs and enqueues them. Workers own the execution lifecycle and update PostgreSQL as processing progresses.

**Retry & Failure Handling**

- Retryable failures (e.g., temporary network or storage issues) update the job to `RETRYING` and are requeued according to the retry policy.
- Permanent failures (e.g., corrupted or unsupported files) mark the job as `FAILED` without retrying.
- If a worker crashes during execution, stale `RUNNING` jobs can be detected and recovered by a separate recovery mechanism or periodic cleanup process.

## Responsibilities

| Component                        | Responsibility                                                                                                                                                                            |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Frontend Client**              | Uploads files directly to the storage bucket, submits processing requests, maintains an SSE connection with the API, displays job status, and downloads processed files.                  |
| **API (FastAPI)**                | Orchestrates the system. Generates presigned upload/download URLs, creates job records, enqueues jobs, authenticates requests (future), and pushes job status updates to clients via SSE. |
| **PostgreSQL**                   | The source of truth. Stores job metadata, current status, execution history, and later user accounts, subscriptions, and analytics.                                                       |
| **Redis Queue**                  | Holds pending `job_id`s and distributes work to available workers.                                                                                                                        |
| **Redis Pub/Sub**                | Carries lightweight events (e.g. `job_completed`, `job_failed`) from workers to the API for real-time notifications.                                                                      |
| **Workers**                      | Consume jobs from the queue, retrieve files from storage, perform processing, upload results, update job state in PostgreSQL, and publish status events.                                  |
| **Storage Bucket (S3/R2/MinIO)** | Stores input and output files. Files are uploaded and downloaded directly by clients or workers using presigned URLs, without passing through the API.                                    |
